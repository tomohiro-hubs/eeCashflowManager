# agent.md

## 目的
- この開発案件での実装品質を安定させる
- 本番反映時のエラーや見落としを減らす
- 社内システムとして直感的で使いやすい画面を優先する

## 基本方針
- 過剰設計を避ける
- 既存構成を尊重し、最小差分で直す
- 新規依存の追加は必要最小限にする
- 見た目より運用しやすさ、分かりやすさ、修正しやすさを優先する

## UI 方針
- 社内利用を前提に、迷わない導線を優先する
- 項目名、ボタン名、状態表示は一読で意味が分かる文言にする
- よく使う操作は近くに置く
- 絞り込み、検索、CSV、編集、保存などの主要操作は見つけやすくする
- 入力エラーはその場で分かるようにする
- 削除、上書き、確定などの操作は確認導線を入れる

## 実装方針
- フォーム、一覧、検索、集計は単純で追いやすい実装を優先する
- 命名は省略しすぎず、役割が分かる名前にする
- 共通化は、重複を減らす効果が明確な場合だけ行う
- 変更前に影響範囲を確認する
- 既存のデータ構造、API、画面挙動を壊さない

## バリデーション
- 入力値はフロントとサーバーの両方で確認する
- 必須、文字数、数値範囲、日付形式、列挙値を明示的に検証する
- 空文字、null、未定義、型ずれを雑に扱わない
- エラー時は握りつぶさず、利用者に分かる形で返す

## 本番事故防止
- 実装後は変更箇所だけでなく、関連画面も確認する
- 条件分岐を足したときは、通常系と異常系の両方を確認する
- フィルタ、検索、保存、更新、削除、CSV、モーダルは副作用を確認する
- 本番反映前に設定値、環境、デプロイ先を確認する
- deploy コマンドは曖昧にせず、対象環境を明示する
- エラー文は原因切り分けに使える粒度で残す

## 作業ルーティン
1. 変更前に対象箇所と影響範囲を確認する
2. 既存実装に合わせて最小差分で直す
3. 入力、表示、保存、更新、削除の影響を確認する
4. 想定エラー時の表示と復帰動作を確認する
5. デプロイ前に設定と対象環境を再確認する
6. 作業後に変更ファイル、理由、確認内容、懸念点を短くまとめる

## 禁止寄りのルール
- 根拠なしに大きくリファクタしない
- 不要なライブラリを追加しない
- 一見きれいでも追いにくい抽象化を増やさない
- 確認なしで本番向け設定を触らない
- 未確認の推測で仕様を決めない

## 優先順位
1. 本番で安全に動く
2. 社内利用で迷わず使える
3. 修正しやすい
4. 実装が短く明快
